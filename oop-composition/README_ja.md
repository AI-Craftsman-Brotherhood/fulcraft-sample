# oop-composition

## 概要

コンポジションによる集約ルート設計を示す注文・明細行サンプルです。FUL の解析対象として、クラス間の所有関係（`Order` が `OrderLine` のリストを保持）と、集約内の責務分散を静的解析・文書化ツールがどう捉えるかを検証できます。

## ソース構成

| 項目 | 値 |
|------|-----|
| ファイル数 | 2 |
| クラス / インターフェース数 | 2（class 2） |
| 公開メソッド数 | 6（`addLine`, `lineCount`, `subtotal`, `containsProduct`, `productName`, `lineTotal`） |
| 総行数（空行含む） | 62 |

## 複雑度の特徴

- `Order.subtotal` と `Order.containsProduct` がそれぞれ `for` ループを含み、ループ内での `OrderLine` メソッド呼び出しによってクラス間依存が発生する
- `addLine` の `null` チェック、`lineTotal` の `unitPrice <= 0 || quantity <= 0` 条件など、ガード節が複数のメソッドに分散している
- `Order` は `List<OrderLine>` をフィールドに持つが、外部に直接公開せず操作メソッドを通じてのみアクセスさせる設計で、カプセル化の強度確認に適している
- 継承階層なし、インターフェース実装なし、という最もフラットな依存構造のため、FUL の解析結果のベースラインとして使いやすい
- `OrderLine` は `Order` を参照せず、依存方向が一方向に統一されている

## FUL 機能との対応

| FUL 機能 | このサンプルでの検証ポイント |
|----------|--------------------------|
| ANALYZE  | クラス間の所有関係（`Order` → `List<OrderLine>`）の依存グラフへの反映。コレクションフィールドの型解析精度 |
| DOCUMENT | 集約ルート `Order` の責務説明と `OrderLine` との関係記述の品質。ループ処理を含むメソッドへの処理フロー説明生成 |
| REPORT   | ループ分岐を含むメソッドのサイクロマティック複雑度スコアの基準値確認。シンプル構造での品質ゲート通過基準の確認 |
