# basic-encapsulation

## 概要

プライベートフィールドへのアクセスをコンストラクタとメソッド経由に限定した `BankAccount` クラスを解析対象とするサンプルである。入力検証・状態変更・boolean 戻り値による操作結果の通知が混在する設計を FUL がどのように解析・文書化するかを検証できる。

## ソース構成

| 項目 | 値 |
|------|-----|
| ファイル数 | 1 |
| クラス / インターフェース数 | 1 |
| 公開メソッド数 | 4 |
| 総行数（空行含む） | 45 |

## 複雑度の特徴

- コンストラクタと各メソッドに null / 範囲チェックの分岐が分散し、クラス全体の合算複雑度が単一メソッドの数値より大きくなるパターン
- `withdraw` は 2 段階の条件チェック（正値確認 → 残高確認）と boolean 返却を持つ
- `final` フィールドと可変フィールド（`balance`）が共存し、Spoon による状態変化解析の対象となる
- 外部クラス依存ゼロ、継承なし

## FUL 機能との対応

| FUL 機能 | このサンプルでの検証ポイント |
|----------|--------------------------|
| ANALYZE  | `final` フィールドと可変フィールドの識別、コンストラクタ内分岐の複雑度計上の正確性確認 |
| DOCUMENT | 副作用（残高変更）を持つメソッドと副作用なし（getter）の違いを LLM が JavaDoc に明示できるかの確認 |
| REPORT   | メソッド単位 vs クラス単位での複雑度集計結果の差異をレポートで確認するユースケース |
